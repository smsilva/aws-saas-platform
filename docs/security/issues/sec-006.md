# SEC-006 — Cluster nodes with IMDSv1 enabled (eksctl default)

| Field | Value |
|---|---|
| **Severity** | Medium |
| **Script** | `scripts/02-create-cluster` |
| **Status** | Open |

## Problem

`eksctl` does not disable IMDSv1 on the node group. With IMDSv1 active, any process inside a pod (via `hostNetwork: true` or through SSRF) can access `http://169.254.169.254/latest/meta-data/iam/security-credentials/` without a session token and obtain the node's temporary IAM credentials.

## Impact

Privilege escalation: node IAM role credentials (`AmazonEKSWorkerNodePolicy`, `AmazonEC2ContainerRegistryReadOnly`) accessible without authentication via SSRF or a compromised pod.

## Fix

Enforce IMDSv2 (token required) on the node group:

```yaml
managedNodeGroups:
  - name: ${cluster_name}-nodes
    # ... other config ...
    instanceMetadataOptions:
      httpTokens: required          # enforces IMDSv2
      httpPutResponseHopLimit: 1    # blocks IMDS access from inside containers
```

With `httpPutResponseHopLimit: 1`, the IMDSv2 token cannot traverse the additional hop of the pod network overlay, blocking IMDS access even with IMDSv2.
